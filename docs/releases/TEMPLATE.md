# JAVM v<version>

## 概览

- 发布范围：`<previous-tag>..HEAD`
- 用 2 到 4 条描述本版本最重要的用户可感知变化。

## 新增

- 写新增能力与新入口，强调使用场景与用户收益。
- 不写纯工程动作（如仅重构、脚本调整）为新增功能。

## 优化

- 写体验、性能、稳定性、流程效率等改进。
- 说明优化前后的差异，避免只有技术实现描述。

## 修复

- 写用户可观察到的问题现象和修复结果。
- 不要只写“修复若干问题”这类泛化表述。

## 兼容性与迁移

- 如有破坏性变更、配置变更、数据迁移，在此明确说明步骤。
- 若无迁移要求，写“无特殊操作要求”。

## 已知问题

- 记录仍存在但可接受的限制、边界条件和临时规避方案。
- 若无明显风险，写“暂无已知问题”。

## 编写要求

- 必须结合提交标题和正文总结，不要只看标题。
- 优先写用户可感知变化，其次再写工程、构建、脚本调整。
- 不要编造未出现的功能、修复或性能数据。
- 不要新增“参考提交”“提交列表”等附录型章节，提交明细仅保留在 `.release-context/`。
- 最终文件命名为 docs/releases/v<version>.md。
